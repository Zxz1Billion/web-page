A working field manual for Integrated Dynamics — a logic and automation network built not on configured GUIs but on Variables, Operators and Aspects. You read the world into variables, compute over them with a visual programming language, and write the results back out as redstone, displays, and item movement. The learning curve is steeper than XNet's, but nothing else in 1.19.2 is this programmable.

00 — Orientation

Start Here

Integrated Dynamics (ID) is a single, powerful logic-and-automation network. Instead of clicking through a per-machine config, you describe what you want as data: Readers turn the world into Variables, the Logic Programmer combines those variables with Operators into derived values, and Writers turn the result back into actions. It has a steeper learning curve than XNet — but it is far more programmable, and a single network can drive logic, displays, and (with the companion mod) full item/fluid/energy logistics.

core

Integrated Dynamics

Cables, parts, Variables, Operators, Aspects, the Logic Programmer, Readers/Writers/Displays, Proxies & Variable Stores.

logistics

Integrated Tunnels

The companion mod: import/export of items, fluids and energy over the same network, with variable-driven predicate filters.

helper

JEI / EMI

Every recipe — Menril resin, crystals, the Logic Programmer, cables and parts — is documented in your recipe viewer. Use it constantly.

vs.

XNet / RS / AE2

ID overlaps with these but trades convenience for raw expressiveness. See the Automation & Power hub for how they compare and combine.

The mental model to learn first

Think of the network as a tiny dataflow program. A value has a type (Boolean, Integer, Item, Fluid, List, …). A Variable Card stores a value. An Operator takes one or more values and returns a new one. Aspects are the network's read/write "ports" to the world. Everything else is wiring.

The shape of the climb

  1. Grow Menril. Find or grow a Menril Tree, then make Menril Resin and Crystalized Menril Crystals in the Squeezer and Drying Basin. These are the backbone of every recipe.
  2. Lay cable. Place a few Cables, attach a Logic Director if needed, and add an energy Battery for the parts that draw power.
  3. Read something. Snap a Reader part onto a cable face touching a chest or redstone, and read an Aspect into a Variable Card.
  4. Compute. In the Logic Programmer, combine variables with Operators to derive what you actually care about (e.g. "is this chest more than half full?").
  5. Write it back. Feed the derived variable into a Writer (emit redstone, set a Display, trigger an action).
  6. Scale with Proxies. Store reusable variables in a Variable Store / Proxy, then reach them anywhere on the network.
  7. Move resources. Add Integrated Tunnels for predicate-driven import/export of items, fluids, and energy.

01 — The raw stock

Menril & Materials

Integrated Dynamics has its own resource: the Menril Tree, a tall blue-tinted tree that generates in cold biomes. Everything in the mod traces back to it. You harvest two products from Menril — a sticky resin and a glassy crystal — and turn them into the cables, cards, and the Logic Programmer.

Menril sources & the two processing machines
MaterialWhere / howUsed for
tree Menril Log / WoodMenril Trees in cold/snowy biomes; replant Menril SaplingsStructural blocks, fuel, planks for early recipes
drop Menril BerriesHarvested from Menril leaves (also a minor food)Squeeze into Menril Resin
block Crystalized Menril ChunkBud-like blocks grown on/near Menril; mined directlySource of Crystalized Menril Crystals via the Squeezer
machine SqueezerCrafted block; you stand on it / power it and drop items inBerries → Menril Resin; Chunks → loose Crystals; ore/material squeezing
machine Drying BasinCrafted block holding fluid + an item over timeResin → solidified Menril; Menril Resin + water/time → crystals & intermediates
Squeezer vs. Drying Basin

The Squeezer is the "crush it now" machine — press items to extract resin, crystals, or fluids. The Drying Basin is the "leave it to set" machine — it holds an item in a fluid and transforms it over time (great for turning liquid/solid Menril into solid crystals). Most early ID recipes need one or both.

The core crafting components

  • Crystalized Menril Block — the decorative/structural form and a key crafting ingredient; the crystals are the "logic glass" of the mod.
  • Logic Programmer — the workbench where you author variables. It pairs a slot for an input Variable Card (or two) with an Operator picker and writes the result back onto a card.
  • Variable Cards — blank cards that store a single value; you stamp them in the Logic Programmer or by reading an Aspect, then slot them into Readers/Writers/Stores.
  • Cables — the network medium; craft a stack early, they are cheap once Menril flows.
  • Squeezer & Drying Basin — keep both near your tree farm; you will run a lot of material through them.
Numbers are approximate

Yields, drying times, and the exact resin/crystal counts vary by pack version and config. Treat anything quantitative in this guide as a guideline and let JEI/EMI and the in-game machine GUIs be the ground truth.

02 — The wiring

The Network

An Integrated Dynamics network is just a connected web of Cables. Parts attach to the faces of those cables and either read from, write to, or display data about the block the face touches. Some parts also draw energy, so a network usually needs a Battery to keep them running.

Cables, connections & the Logic Director

Cables auto-connect to neighbours. Use a Wrench to disconnect a side, route around things, or pop a part off. The Logic Director is the handheld that lets you copy and paste variable references and inspect what a face is exposing — it is your debugger when a value "isn't arriving" where you expect.

Network ingredients
Block / itemRole
CableThe network medium. Connect them into one continuous web; parts live on their faces.
Battery (energy)Supplies the power some parts consume per tick. Charge it from an ID generator or any FE source feeding it.
Logic DirectorHandheld tool to copy/paste variable references between cards and parts; inspect network exposure.
WrenchPlace/remove parts, toggle a cable side's connection, rotate displays.
Coal Generator / other energyOptional power source; only needed once you run energy-hungry parts at scale.
Do you need a Battery?

Plain logic (a Reader feeding a Writer through one Operator) often costs little or nothing. But Displays, complex Operators, and especially Integrated Tunnels import/export draw energy per tick. Once parts start "blinking off," add a charged Battery to the network and feed it power.

Part attachments — how parts touch the world

A part is a small device that clips onto a cable face. There are four families, and they are the whole point of the mod:

read

Readers

Expose the adjacent block's state as variables — inventory contents, redstone level, fluid amount, machine progress, block identity, and more.

write

Writers

Consume a variable to act on the adjacent block — emit a redstone signal, set NBT, trigger a capability.

show

Displays

Render a variable's current value as floating text/icon on the cable face, in-world. Pure read-out, no action.

sense

Detectors

Specialised readers that watch for a condition (e.g. an item entering an inventory) and surface it as a boolean for downstream logic.

The same cable face can host one part. A Reader exposes Aspects (§05) that you capture into Variable Cards; a Writer accepts a card and applies it; a Display just mirrors a value. This read → compute → write loop is the entire programming surface of Integrated Dynamics.

03 — The language

Variables & the Logic Programmer

This is the heart of the mod. A Variable Card holds one typed value. The Logic Programmer is where you build new variables by feeding existing ones through Operators. It is a small, strongly-typed, visual programming language — and once it clicks, you can compute almost anything about your base.

Values have types

Common value types (the data ID flows around)
TypeExample valueWhere it comes from
Booleantrue / falseRedstone state, comparisons, "is full" checks
Integer42Item counts, redstone strength, fluid mB, tick timers
Double0.75Fill ratios, energy fractions, computed averages
String"diamond"Names, labels, formatted display text
Item Stack64× Iron IngotReading an inventory slot or a held item
Fluid Stack1000 mB LavaReading a tank's contents
Block / Entitya furnace, a cowReading the adjacent block or nearby entities
List[a, b, c]A whole inventory, a set of entities, mapped results
Operator(a function value)Operators are first-class — you can pass them to map/filter
Types must match

The Logic Programmer is strict: an Operator that wants two Integers will refuse a String, and the card stays red until the types line up. This is a feature — it catches mistakes before they reach the world. When a build "does nothing," check that every card is green and the types match end to end.

Operators — the verbs

Operators take one or more values and return a new one. The Logic Programmer groups them; the families you will use constantly:

Operator families
FamilyExamplesTypical use
Arithmetic+, −, ×, ÷, mod, min, maxCompute totals, fill ratios, thresholds
Logic / ComparisonAND, OR, NOT, =, >, <, ≥Combine conditions into a single boolean
Listlength, get, append, map, filter, reduceProcess a whole inventory or entity set at once
Item / NBTitem name, count, NBT tag get, durability, is-stackableInspect stacks, match by NBT, sort by computed key
Fluid / Block / Entityfluid amount, block hardness, entity health, is-aRead deeper world state for predicates
Conditionalif-then-elseBranch a value based on a boolean
Operators are values too

One of ID's most powerful ideas: an Operator is itself a value. You can store a function on a card and feed it to map or filter over a list — for example, "filter this inventory to stacks whose count > 16" is one list operator plus one comparison operator partially applied. This is how you build computed predicates for sorting.

Authoring a variable, step by step

  1. Read or place inputs. Capture world state onto a Variable Card via a Reader's Aspect, or place a constant value with the Logic Programmer's input cards.
  2. Pick an Operator. Open the Logic Programmer, choose the Operator that matches your inputs' types.
  3. Wire the inputs. Drop your source cards into the Operator's input slots. The output preview turns green when types match.
  4. Write the result. A blank card in the output slot becomes a new derived Variable Card holding the computed value.
  5. Reuse it. Slot that card into a Writer, a Display, or a Proxy — or feed it as input to the next Operator.

04 — The I/O

Readers & Writers

Variables only matter if they connect to the world. Readers are your inputs, Writers are your outputs, and Displays are the human-readable middle. All three are parts that clip onto a cable face pointing at the block you care about.

Readers — world → variable

A Reader exposes a set of Aspects for whatever it is pointed at. Point one at a chest and you get inventory Aspects; point one at a redstone source and you get redstone Aspects; point one at a machine and you get its progress/state where supported.

Reader types & what they surface
ReaderReadsSample Aspects
Inventory ReaderAdjacent inventorySlot count, item in slot N, total count, fullness ratio, is-empty
Redstone ReaderRedstone state of the sideSignal level (Integer), powered? (Boolean), comparator value
Fluid ReaderAdjacent tankFluid amount (mB), capacity, fluid type, fill ratio
Machine / Energy ReaderCapability-exposing blockStored energy, max energy, progress, is-working
Block / World ReaderThe adjacent block & surroundingsBlock identity, light level, biome, time of day, weather
Entity ReaderNearby entitiesEntity list, counts, health, type predicates

Writers — variable → world

A Writer takes a Variable Card and applies its value to the adjacent block through a matching Aspect. The classic is the Redstone Writer: feed it a Boolean or Integer and it emits that as a redstone signal. Others set display text, toggle machine inputs, or write NBT where supported.

Writer types & what they do
WriterConsumesEffect on the world
Redstone WriterBoolean / IntegerEmits a redstone signal of that strength out the face
Display WriterAny valueSets the text/icon shown by a Display part
Fluid / Energy WriterIntegerWhere supported, sets transfer/target amounts on the block
NBT / Capability WriterMatching typeWrites a value into a block's exposed capability/NBT field
Displays show, Writers act

A Display renders a variable in-world (e.g. "Power: 73%" floating on the cable) but changes nothing. A Writer changes the world. Wire a Display alongside any non-trivial Writer so you can see the value the Writer is acting on — it makes debugging enormously faster.

One face, one part

Each cable face holds a single part. If you need to both read a chest and emit redstone next to it, use two faces (or two cables). Plan your cable runs so the faces you need are actually adjacent to the right blocks.

05 — Capabilities & reuse

Aspects & Proxies

Two concepts make ID scale: Aspects (the typed read/write capabilities a part exposes) and Proxies / Variable Stores (places to park a variable so the whole network can reuse it).

Aspects — the network's ports

An Aspect is a single named capability on a part: "the redstone level on this side," "the item in slot 3," "is this tank full." Read Aspects produce values; write Aspects consume them. When you capture a Reader's Aspect onto a Variable Card, you are subscribing that card to a live world value — it updates as the world changes.

How an Aspect flows
StageWhat happens
ExposeA Reader/Writer part offers a list of Aspects appropriate to its target.
CaptureYou bind a read Aspect to a Variable Card — the card now mirrors that live value.
ComputeOperators in the Logic Programmer transform the captured value into a derived one.
ApplyA write Aspect on a Writer consumes the final card and acts on the world.

Proxies & Variable Stores — reuse across the network

Re-deriving the same value in twenty places is wasteful and error-prone. A Variable Store holds named variables; a Proxy lets any part on the network reference a stored variable by its card, no matter where it physically lives. Compute "is the main bus over 50% full?" once, store it, and read it from every Writer that needs it.

Reuse blocks
BlockRole
ProxyA reference handle: point many parts at one shared variable. Edit the source once, every consumer updates.
Mechanical Variable StoreBulk storage for many Variable Cards the network can pull from; the "variable library" of a base.
Menril Variable StoreCrystalized-Menril variant store; same idea, fits the ID aesthetic and recipe tree.
Materialized variables"Freeze" a computed value into a constant card when you don't need it live — cheaper and stable.
Proxies are the antidote to spaghetti

The single biggest quality-of-life upgrade in an ID base is moving shared logic into a Variable Store and referencing it through Proxies. You get one source of truth, far fewer cards to maintain, and a network that's actually readable six months later.

06 — Moving resources

Integrated Tunnels

Integrated Dynamics computes and signals beautifully, but base ID doesn't move items for you. Integrated Tunnels is the companion mod that adds import and export of items, fluids, and energy over the very same cable network — and crucially, it lets you gate every transfer with a variable-driven predicate.

Interfaces & channels

Tunnels adds Interface parts (importers/exporters) for each resource kind. An importer pulls from the adjacent block into the network's transfer; an exporter pushes into the adjacent block. Channels keep multiple transfers from interfering — assign matching channel numbers to the endpoints that should talk to each other.

Integrated Tunnels parts
PartMovesDirection
Item Importer / ExporterItem stacksPull into / push out of the network
Fluid Importer / ExporterFluid (mB)Pull into / push out of the network
Energy Importer / ExporterForge EnergyPull into / push out of the network
Interface (per kind)The endpoint on a block faceBoth, depending on configuration

Predicate filters — "move X only when Y"

This is the killer feature. An exporter doesn't just move "iron"; you hand it an Operator that returns a Boolean per candidate stack. "Export this item only if the destination has fewer than 64 of it"; "import fluid only while the source tank is over 50%"; "push energy only when the buffer is below threshold." The predicate is just another variable you authored in the Logic Programmer.

  • predicate: count < 64
  • predicate: NBT damage = 0
  • predicate: fluid ≥ 50%
  • predicate: name contains "ingot"
  • predicate: energy < threshold
    • Why this beats a plain filter

    A normal pipe filter matches a fixed item list. A Tunnels predicate is computed — it can depend on the destination's current contents, an item's NBT, the time of day, or any other variable on the network. That turns "sort items" into "sort items by a rule I wrote," which is the whole reason to use ID for logistics.

    Mind the energy and channels

    Tunnels transfers consume network energy per operation — keep a charged Battery feeding the network. And when two unrelated transfers seem to fight, it's almost always a channel collision: give each logical route its own channel number.

    07 — In practice

    Example Builds & Tips

    Three small, concrete builds that show the read → compute → write loop end to end, then a set of habits that keep a growing network sane.

    1 — Conditional auto-crafting stock-keeper

    Goal: keep at least 32 steel plates in a buffer chest, crafting more only when stock drops.

    1. Read. Inventory Reader on the buffer chest → Aspect "count of steel plates" → Variable Card (Integer).
    2. Compute. Logic Programmer: comparison Operator count < 32 → derived Boolean card.
    3. Write. Redstone Writer fed the Boolean → emits redstone to start your crafter/factory when stock is low.
    4. Watch. Add a Display showing the current count so you can confirm the threshold at a glance.

    2 — Redstone clock from variables

    Goal: a clean on/off pulse without a vanilla repeater loop.

    1. Read. A World/Block Reader exposes a tick or time-of-day value → Integer card.
    2. Compute. (time mod N) < H → Boolean that is true for H ticks out of every N.
    3. Write. Redstone Writer emits the Boolean — adjust N and H to set period and duty cycle precisely.

    3 — Item sorting by a computed predicate

    Goal (with Integrated Tunnels): route only "valuable" items to a vault, everything else to a dump.

    1. Author the predicate. An Operator value: "is this stack's name in my valuables list" or "is its NBT-derived score > X".
    2. Export to vault. Item Exporter toward the vault, filtered by that predicate.
    3. Export the rest. A second Exporter toward the dump using the negated predicate (NOT).
    4. Channel them. Give the two routes distinct channels so they never compete for the same stack.

    Tips for taming the complexity

    habit

    Build incrementally

    Get one Reader → one Operator → one Writer working before adding the next stage. Verify each link with a Display before chaining further.

    habit

    Label your variables

    Name every card for what it means ("steel<32"), not what it is. Future-you needs the intent, not the type.

    habit

    Use Proxies for shared logic

    Compute a value once, store it, reference it via Proxies everywhere. One source of truth beats twenty copies.

    habit

    Display while debugging

    Sprinkle Display parts on intermediate values. Seeing the number is faster than guessing why a Writer is silent.

    When in doubt, check the types

    Almost every "it does nothing" bug in Integrated Dynamics is a red card: a type mismatch, an empty input slot, or a Proxy pointing at the wrong variable. Walk the chain from Reader to Writer and confirm every card is green and labelled.

    Power it before you scale it

    A two-part demo runs on nothing, but Displays, heavy Operators, and Integrated Tunnels transfers all draw energy. Before you expand a working prototype across the base, put a charged Battery on the network and a real power source behind it.

    ★ — The path

    Progression — from first variable to a logic brain

    Every other section explains a piece of Integrated Dynamics in isolation. This is the order to actually build them in. Follow it once end to end and you will have a network that reads the world, computes over it, and acts back on it — the foundation every later automation bolts onto. Resist the urge to wire ten parts at once; ID rewards the patient builder who verifies each link before adding the next.

    1. Gather Menril & craft the core. Find or grow a Menril Tree, run berries through the Squeezer for Menril Resin, and set Crystalized Menril Chunks in the Drying Basin for crystals. From those, craft a stack of Cables, a Logic Programmer, and a handful of blank Variable Cards. That kit is everything you need to start programming.
    2. Lay cable and a Reader → Display. Run a short cable web, keep a Logic Director in hand for inspecting and copy/pasting variable references, and clip an Inventory or Redstone Reader onto a face touching a chest or a lever. Add a Display part and point it at the read value. Seeing a live number prove the network works is the moment ID clicks.
    3. Open the Logic Programmer and combine variables. Capture the Reader's Aspect onto a card, then in the Logic Programmer feed it through an Operator — a comparison like count < 32 or an arithmetic fill ratio. The output card turns green when the types line up, and you now hold a derived variable that means something, not just raw state.
    4. Add a Writer that acts on the world. Feed the derived Boolean into a Redstone Writer so the network emits a signal — start a crafter, flip a door, gate a machine. This closes the read → compute → write loop for the first time: the world changed because of a rule you authored.
    5. Reuse with Variable Stores & Proxies. Once two parts want the same value, stop re-deriving it. Park computed variables in a Variable Store and reference them through Proxies anywhere on the network. One source of truth means fewer red cards and a base you can still read months later.
    6. Bolt on Integrated Tunnels. With the companion mod, add Item / Fluid / Energy Importers and Exporters and gate each transfer with a variable-driven predicate — "export only if the destination has fewer than 64" or "import only while the source tank is over 50%." Assign distinct channels so unrelated routes never fight, and keep a charged Battery on the network.
    7. Ship a worked conditional automation. Combine the lot into a real machine: a keep-N-in-stock loop (Inventory Reader → count < N → Exporter or crafter trigger) or a redstone clock ((time mod N) < H → Redstone Writer). Now you have a logic brain that maintains stock or paces a process on its own.
    Two habits make all of this stick

    Build incrementally — get one Reader, one Operator, one Writer working before adding the next stage, and verify every link with a Display. And label every variable for what it means ("steel<32"), not what it is. From here, the Automation hub shows how ID fits beside XNet, RS, and AE2, and The Ascent places this network on the wider modpack climb.
