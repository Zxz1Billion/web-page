The Control
Bus

A control engineer's handbook for the automation and power-network mods in To the Moon 2.0. Four networking philosophies share one base: XNet routes everything cheaply, Integrated Dynamics computes and conditions, Integrated Tunnels moves matter over that same logic fabric, and Flux Networks deletes your power cables entirely. Underneath sit two foundations — RFTools Base and the self-contained IC2 Classic tech tree — and a chapter on making FE, Joules and EU talk to each other.

00 — Orientation

Start Here — the landscape

This pack ships several automation toolkits that solve overlapping problems in very different ways. Pick the right tool per job and your base stays clean; mix them blindly and you fight your own wiring. Here is the mental model.

Four broad approaches live side by side. They are not rivals so much as different layers you reach for at different moments:

routing

XNet

A cable network whose only job is to move items, fluids, energy and logic between blocks with simple per-connector rules. Cheap, fast, the everyday workhorse.

logic

Integrated Dynamics

A programmable network: read world state into typed variables, compute with operators, and drive displays, detectors and writers. Automation that thinks.

transfer

Integrated Tunnels

Adds item/fluid/energy import & export to an Integrated Dynamics network, gated by the same variables and predicates. Conditional logistics.

power

Flux Networks

Pure wireless FE. Plugs feed, Points draw, Storage banks — across any distance, no cable runs. Replaces a power grid with three block types.

Which tool for which problem?

Reach for the simplest thing that solves the job in front of you:

A decision table for automation problems
You want to…Best toolWhy
Move items/fluids/energy between a handful of machinesXNetOne controller, cheap connectors, per-side insert/extract/filter GUIs. No programming.
Run a machine only when a condition holdsIntegrated DynamicsRead state → boolean variable → drive a redstone Writer or an Integrated Tunnels gate.
Show counts, fill levels or status on a wallIntegrated Dynamics (Display)Variables render live to a Display Panel; numeric & text formatting built in.
Import/export with fine item/fluid predicatesIntegrated TunnelsFilter by NBT, tag, count thresholds and arbitrary operator expressions.
Deliver power base-wide with zero cablesFlux NetworksWireless FE with per-network access, limits and priorities. Distance-agnostic.
Double ores & climb a self-contained tech ladderIC2 ClassicEU economy, Macerator ore-doubling, voltage tiers — a whole progression in one mod.
The other guides own their own logistics

Mekanism (Logistical Transporters / QIO), Applied Energistics 2 (ME network) and Create (belts, funnels, trains) each have a dedicated guide and their own internal logistics. This page covers the general-purpose networks that glue everything else together — and the standalone IC2 Classic tech tree.

A silent dependency

Titanium (by Buuz135) is a library/framework mod several mods in the pack lean on. You never craft anything from it — but if a dependent mod misbehaves, confirm Titanium is present and version-matched. See §08.

01 — The everyday router

XNet — one controller, infinite plumbing

XNet (by McJty) is the workhorse of this pack: a single cable network that hauls items, fluids, energy and logic with nothing more than a Controller and some cable. It is deliberately cheap and extremely fast — most small-to-medium bases never need anything else.

The pieces

brain

Controller

The single block that runs the network. Holds the channel definitions and is where all the routing GUIs live. One controller per network.

cable

Network Cable

The backbone. Carries the abstract "channels" — it does not itself transport contents, it links connectors to the controller.

port

Connector

Where the network touches an inventory/tank/machine. Each side of a connector hosts insert/extract rules. Advanced connectors raise throughput.

wireless

Antenna

Bridges two network segments wirelessly via a Wireless Router + Antenna, so distant rooms join one controller without running cable.

Channels — the core idea

A network exposes a fixed number of channels. Each channel has a type and is configured once, then connectors are subscribed to it. The four channel types each open a tailored GUI:

XNet channel types
ChannelMovesPer-connector controls
ItemItems between inventoriesInsert / extract mode, filter (whitelist/blacklist + tag/meta/NBT), count limits, speed, priority, round-robin.
FluidTanks & fluid handlersInsert/extract rate, per-fluid filter, min/max levels.
EnergyFE/RF between buffersPer-tick transfer cap, min/max levels to gate charging.
LogicBoolean signals + sensorsSensors that read inventory/fluid/energy/redstone state into colored logic signals that drive outputs elsewhere.

How a transfer is wired

  1. Place a Controller; run Network Cable from it past every machine you want to touch.
  2. Replace the cable face touching a machine with a Connector (or place one and connect it).
  3. Open the Controller, create a channel (say, Item), and name it.
  4. For each connector on that channel, set a side to extract (source) or insert (destination), add filters and a speed.
  5. The controller round-robins or prioritizes deliveries from extractors to inserters every tick.
The Logic channel is underrated

XNet's Logic channel can read "inventory contains ≥ N of X" or "tank below 20%" and emit a colored signal that another connector consumes as a redstone-style condition. For simple "turn the pump off when the tank is full" automation you often don't need Integrated Dynamics at all.

Channels are the budget, not cable

Cable is effectively free; the scarce resource is the channel count per controller and the connector tier (basic vs advanced) which caps per-tick throughput. If a line can't keep up, upgrade the connector or split onto a second controller — don't just add cable.

02 — The logic network

Integrated Dynamics — automation that computes

Integrated Dynamics (by kroeser) is a different animal from XNet: instead of routing contents, it builds a typed logic network. You read facts about the world into variables, transform them with operators, and feed the results into displays, detectors and writers. It is, in effect, a visual programming language wired into your base.

Bootstrapping — Menril & the crafting parts

Before any logic, you need the mod's signature material. Integrated Dynamics introduces Menril trees (a tall blue-leaved tree) whose products you process into the core crafting reagents:

  • Menril Wood / Logs — base building blocks for the network parts.
  • Menril Berries & the Squeezer — squeeze inputs to extract Menril Resin and other liquids.
  • Drying Basin — dries Menril Resin (and other recipes) over time into Crystalized Menril blocks/chunks, the substance most logic components are crafted from.
  • Crystalized Menril Chunks — the "silicon" of the mod: variable cards, the Logic Programmer, parts and the network's connective Cables all trace back to it.
Squeezer + Drying Basin

Both are manual-or-automatable single-block machines. The Squeezer can be hand-cranked early; the Drying Basin simply needs time. Build one of each near your Menril grove and you have the full component supply chain.

The network fabric

backbone

Cables

Logic Cables form the network. Parts attach to the faces of blocks adjacent to a cable; everything on one connected cable run shares state.

tool

Logic Programmer

The bench where you author Variables: pick a value, an operator, or compose existing variables into a new one. Outputs a Variable Card.

storage

Variable Store

Holds the Variable Cards your Readers/Writers reference. The network's "memory".

power

Energy Battery

Logic parts draw a trickle of energy; an Energy Battery (or a Coal/Mechanical source) keeps the network powered.

Aspects, Operators & Variables

The whole system is built on a strict type system. Three concepts do the work:

The Integrated Dynamics type system
ConceptWhat it isExample
AspectA specific readable/writable property a part exposes"Inventory: item count in slot 0", "Fluid: amount", "Block: is powered"
VariableA typed value held on a card — boolean, integer, string, list, item, fluid, etc.A card holding the integer 64, or a list of ItemStacks
OperatorA function combining variables into a new value>=, +, length(list), and, if/then/else

You build logic by composing: a Reader exposes an Aspect as a variable, you feed it through Operators in the Logic Programmer to produce a derived variable, then a Writer or Display consumes the result.

The parts that read, write & show

Core part types
PartRoleTypical use
ReaderExposes Aspects of an adjacent block as variablesRead a machine's energy %, an inventory's contents, redstone level, world time.
WriterWrites a variable's value into the world/blockEmit a redstone signal, set a display, trigger another machine.
Display (Panel)Renders a variable visually on a panelLive item counts, fluid bars, on/off status, formatted text.
DetectorMeasures network/cable state (e.g. variable presence)Conditional branching based on whether a signal exists.
Logic DirectorCopies/relays variables across cable boundariesMove a computed variable from one sub-network to another cleanly.

Worked example — a conditional pump

  1. Reader on a tank exposes the fluid amount Aspect as an integer variable.
  2. In the Logic Programmer, compose fluidAmount < threshold with a < operator and a constant → a boolean variable card.
  3. Store the card; a Writer reads it and emits a redstone signal when true.
  4. That signal switches the pump on only while the tank is below the threshold — hysteresis and all.
Build boolean and numeric networks freely

Because operators chain, you can express almost any condition: "run the smelter only at night AND while the furnace buffer is below 80% AND there's coal in the chest". Integrated Dynamics is the answer whenever the question contains the word "only when".

03 — Logic-gated transfer

Integrated Tunnels — moving matter over logic

Integrated Tunnels (also by kroeser) is the add-on that gives an Integrated Dynamics network real logistics. It adds Importers and Exporters for items, fluids and energy that run over the same cables — but unlike XNet, every transfer can be gated by the network's variables and arbitrary predicates.

Importers & Exporters

These are network parts, mounted like Readers/Writers on cable-adjacent faces. Each comes in item, fluid and energy flavours:

Tunnels transfer parts
PartDirectionCarries
Item Importer / ExporterInto / out of the networkItemStacks between inventories
Fluid Importer / ExporterInto / out ofFluids between tanks/handlers
Energy Importer / ExporterInto / out ofFE/RF between buffers
Interface (channel)Network-internal routingTag transfers with a channel id to keep flows separate

Fine-grained filters & predicates

This is where Tunnels earns its keep. An Importer/Exporter's filter slot accepts not just an item template but a variable — including a predicate built in the Logic Programmer. That means a transfer rule can be a full boolean expression:

  • Match by exact stack, item tag, mod, NBT, or "any item where damage < 10%".
  • Rate & quantity bound to a variable — e.g. export only enough to top a chest to exactly 64.
  • Conditional enable — wire the whole exporter to a boolean variable so it only runs when your Integrated Dynamics logic says so.
  • Channels keep independent flows from colliding on the same cable run.
Tunnels vs XNet — pick by complexity

For "drain this chest into that machine", XNet is faster to set up. The moment your filter needs "items matching a computed predicate" or "export only while condition X holds", Integrated Tunnels is the tool — it inherits the entire Aspect/Operator type system from Integrated Dynamics.

It needs the same network & power

Tunnels parts live on Integrated Dynamics cables and draw the same trickle of energy. Keep an Energy Battery on the network or transfers will throttle/stall. Predicate evaluation also adds a little overhead — keep expressions tidy on big bases.

04 — Wireless power

Flux Networks — ditch the power cables

Flux Networks (by sonar_sonic) does one thing supremely well: it transmits Forge Energy wirelessly, anywhere, instantly. You never run a power cable again. Generators feed the network at one point; machines draw from it across the base, or across dimensions, with no transmission blocks in between.

The four block types

brain

Flux Controller

Optional but powerful: one per network. Unlocks network-wide control, wireless charging of items, and configuration. A network can run without it but gains features with it.

input

Flux Plug

Attach to a generator or energy buffer to pull power into the network. This is your input node.

output

Flux Point

Attach to a machine to push power out of the network into it. This is your output node.

bank

Flux Storage

A wireless energy bank (Basic → Herculean tiers) that stores the network's surplus. Acts as both buffer and battery.

Networks are owned, named & access-controlled

Each Flux block is bound to a network you create — identified by a name and color, and owned by a player. This makes power both convenient and secure:

Per-network settings
SettingEffect
Owner & access levelPublic / restricted networks; grant other players user/admin access for shared bases.
Name & colorHuman-readable identity; run multiple isolated networks (e.g. "Main", "Reactor").
Transfer limit (per connection)Cap how much FE a given Plug/Point moves per tick — protect fragile machines or ration output.
PriorityOrder in which Points draw and Plugs feed — fill critical machines first, dump surplus to storage last.

A minimal base-wide grid

  1. Create a network in the Flux item/Controller GUI; name and color it.
  2. Slap a Flux Plug on each generator or your main energy buffer — power now flows in.
  3. Slap a Flux Point on each machine that needs power — it draws out, no cables.
  4. Add Flux Storage to bank surplus; set priorities so machines fill before storage.
  5. Optionally place a Controller to wirelessly charge tools and unlock full network stats.
The end of cable spaghetti

Flux is pure FE/RF. Pair it with anything that speaks Forge Energy — Mekanism (via Joule conversion), RFTools machines, Create's stress-to-FE adapters, IC2 via an EU↔FE converter (see §07). Generate anywhere, consume everywhere.

Mind the per-connection limit

A Point's default transfer limit may bottleneck a hungry machine; raise it in the network GUI. Conversely, set a Plug's limit to avoid draining a small generator faster than it can produce.

05 — The shared foundation

RFTools Base — McJty's machine framework

RFTools is not one mod but an ecosystem (McJty), split into modules — Power, Storage, Builder, Utility, Control and more. RFTools Base is the shared library and machine framework they all sit on. On its own it ships little to "play with", but it defines the patterns every RFTools module reuses.

What Base provides

framework

Machine framework

The common code for RFTools machines: energy handling, GUIs, redstone modes, and the modular-block conventions every module inherits.

crafting

Machine Frame

The shared crafting component at the heart of most RFTools machines — the recognizable "frame" block recipes build around.

tool

Smart Wrench

The configuration multi-tool: rotate machines, set sides, and tune RFTools blocks. The standard "screwdriver" of the ecosystem.

shared

Common items

Base materials, dimensional shards/parts and registry plumbing that downstream modules (Power Cells, Storage Scanner, Builder, etc.) depend on.

The RFTools approach

Across modules, RFTools favours modular, configurable machines with clean GUIs and heavy redstone integration:

  • FE-native — everything runs on Forge Energy; it slots straight into a Flux network or any FE grid.
  • Redstone-aware — machines expose redstone modes (ignore/on/pulse), making them easy targets for Integrated Dynamics Writers or XNet logic.
  • Modular blocks — modules like Power Cells form multiblock-ish banks; Storage Scanners index many inventories at once; the Builder quarries and moves blocks.
  • Smart Wrench everywhere — one tool configures the lot.
Why it's listed as "Base"

If the pack includes RFTools Power Cells, the Crafter, the Builder or the Storage Scanner, they each require RFTools Base. Treat Base as the runtime/library; the gameplay lives in the modules built on top of it. Consult JEI for exactly which RFTools modules are installed.

06 — A self-contained tech tree

IC2 Classic — the EU economy

IC2 Classic (maintained by sfPlayer1) is the classic IndustrialCraft experience: a complete, self-contained tech tree built on EU (Energy Units) and a hard voltage system. It does not use Forge Energy internally — EU has its own rules, and one of them will absolutely break your machines if you ignore it.

Overvoltage destroys machines

Every IC2 machine has a voltage tier. Feed it power above its tier and it explodes. A basic machine fed High Voltage is gone. Match your cabling and transformers to the machine, or use a Transformer to step down before the wire reaches it. This is the single most important IC2 rule.

Voltage tiers

EU voltage tiers (packet-based; values are the classic tier ceilings)
TierNameApprox. EU/packet ceilingNotes
LVLow Voltage≈ 32Early machines: Macerator, Electric Furnace, basic generators.
MVMedium Voltage≈ 128Mid-tier machines & better cabling; step down to LV for old machines.
HVHigh Voltage≈ 512Heavy machines & long-distance transmission.
EVExtreme Voltage≈ 2048Top tier; demands the best cables and careful transformers.

Numbers are approximate classic tier ceilings — verify exact values in-game; configs and IC2 Classic specifics may differ.

Cabling & transformers

  • Cables are rated. Each cable type tolerates up to a certain voltage; push more through and the cable (and what's downstream) suffers. Higher tiers also lose less energy per block of distance.
  • Transformers step voltage up/down. An LV Transformer turns one MV packet into several LV packets (and vice-versa). Place a step-down transformer before any machine that can't take the incoming tier.
  • Insulation reduces shock/loss on some cables — relevant for routing near the player.

The early-game ladder

  1. Generator — burn fuel to produce EU and charge batteries. Your first power.
  2. RE Battery — portable EU storage; charges in a generator/charger and powers tools and the early grid.
  3. Macerator — grinds ore into 2 dust: classic ore doubling, the reason to build IC2 early.
  4. Electric Furnace — smelts faster and cheaper than fuel; feed it the macerated dust.
  5. Climb to MV/HV machines, better generators (geothermal, etc.) and transformers as your EU budget grows.
EU is not FE

IC2 Classic's EU is a separate currency with packet-based transmission and voltage — it will not directly accept FE from a Flux network or RFTools. To bridge the two economies you need a converter; see §07. Treat IC2 as its own walled garden until you deliberately bridge it.

07 — Making them talk

Power Interoperability — FE, Joules & EU

The pack hosts three energy economies. Two are nearly the same; one is a stranger. Knowing which is which saves you a lot of confused wiring.

The three energy currencies
CurrencyUsed byInterop
FE / RF (Forge Energy)RFTools, Flux Networks, XNet, most Forge tech, Create adaptersThe lingua franca. FE and RF are interchangeable here.
Joules (J)MekanismAuto-converts to/from FE at a configured ratio — Mekanism cables and machines accept FE transparently.
EU (Energy Units)IC2 ClassicSeparate. Packet-based with voltage; needs an explicit converter block to bridge to FE.

The practical rules

  • FE ⇄ Joules is free. Mekanism speaks FE on its borders. A Flux Point on a Mekanism machine, or FE from RFTools into an Energized Smelter, "just works" — Mekanism converts internally at the config ratio. Treat Joules as FE with a different label.
  • FE ⇄ EU needs a bridge. IC2 Classic's EU will not flow into or out of an FE grid on its own. Use a dedicated energy converter (an Energy Bridge / converter block, if present in the pack) to translate between an EU tier and an FE buffer. Check JEI for the available converter.
  • Mind the conversion ratios. Cross-economy conversions are config-defined and not always 1:1 — a converter may apply a ratio or a small loss. Don't assume parity; read the converter's GUI.
  • Voltage still applies on the EU side. When converting FE→EU, the converter outputs at a specific EU tier. Step it down with a transformer before feeding low-tier IC2 machines, or you'll overvolt them (see §06).
Keep economies in lanes

The cleanest bases keep one FE backbone (Flux Networks) for everything Forge-Energy-native, and treat IC2 Classic as an island that you bridge at exactly one converter. Converting EU all over the base invites overvoltage accidents and ratio confusion.

08 — Field doctrine

Recommended Patterns

Putting it together: a few battle-tested layouts that keep bases readable as they scale. Match the network to the base size and the problem's complexity, not to fashion.

Match the tool to the scale

Doctrine by base size & need
SituationRecommended stack
Small / early base, simple "move A → B"XNet for all item/fluid/energy routing. One controller covers a starter base. Use the Logic channel for basic on/off automation.
Conditional / "only when" automationIntegrated Dynamics for the logic; Integrated Tunnels for any transfer that must obey a predicate. Drive XNet or machine redstone from its Writers.
Base-wide powerFlux Networks. One FE network, generators on Plugs, machines on Points, surplus to Storage. Forget cables.
Big, mixed baseFlux for power + XNet for bulk routing + Integrated Dynamics for the brain. Each does what it's best at; none does everything.
Self-contained tech progressionIC2 Classic as its own EU island; bridge to the FE grid at a single converter when needed.

Three canonical builds

starter

The XNet starter

One Controller, cable past every machine, connectors with extract/insert rules. Auto-feed furnaces, drain a quarry chest, charge a battery bank — all without a single logic gate.

brain

The Int. Dynamics brain

A Menril-fed logic core reading tank levels, machine energy and time, computing booleans, and switching pumps/smelters via Writers. Add Integrated Tunnels for predicate-gated exports.

grid

The Flux backbone

One named/colored FE network spanning the whole base (and dimensions). Plugs on every generator, Points on every machine, Storage banking surplus, priorities protecting critical loads.

Layering them

  1. Power layer: build the Flux network first so every machine has FE wherever it sits.
  2. Routing layer: lay XNet for the bulk item/fluid movement that doesn't need conditions.
  3. Logic layer: add Integrated Dynamics where automation must decide, and Integrated Tunnels where a transfer must obey that decision.
  4. Bridge: keep IC2 Classic as a discrete EU island and bridge it to FE at one converter only.
Titanium — the silent dependency

Titanium (Buuz135) is a library mod that several pack mods depend on. You will never see it in a recipe, but it must be present and version-matched for its dependents to load. If a dependent mod crashes or vanishes from JEI, verifying Titanium is your first check.

When in doubt, simplest wins

If XNet solves it, don't reach for Integrated Dynamics. If a Flux Point delivers power, don't run cable. Complexity is a cost you pay in debugging — spend it only where the problem genuinely needs a brain.
